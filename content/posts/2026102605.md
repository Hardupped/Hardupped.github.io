## 关于避坑清单——实打实踩过的坑

接入过程中我遇到了几个典型问题，写出来帮你绕开：

### 坑 1：企业微信消息长度限制

企业微信单条消息最长 2048 字节（含表情符），如果模型返回内容超长，会直接截断或报错。

**解决方案：** 在调用千聚 API 时，加上 `max_tokens` 参数控制输出长度，或者设置 `temperature` 为 0.3 让回答更简洁。我一般设 `max_tokens=800`，刚好够用。

### 坑 2：流式输出处理不当

企业微信的回调接口支持流式推送（chunk 模式），但很多开发者在首次接流式时，忘了设置 `stream=True` 或忘记用 `for chunk in response` 逐条处理。

**解决方案：** 如果不需要流式，直接设 `stream=False`，获取完整回复再返回。第一次跑通后，再优化为流式体验。

### 坑 3：API key 权限范围

企业微信后端如果有多环境（开发、测试、生产），建议在千聚后台生成多个 key，分别标注用途。别把所有环境共用一个 key，不然出问题时不方便定位。

千聚的 key 支持自定义名称，我建了三个：dev-key、test-key、prod-key，方便管理。

### 坑 4：模型选择不合适

有些企业直接选 GPT-4 做所有任务，成本高且响应慢。企业微信内部问答很多是基础知识、流程咨询，用 gpt-4o-mini 或 DeepSeek-V3 就足够。

**避坑建议：** 先评估场景复杂度。简单问答用 gpt-4o-mini（价格低、速度快），复杂推理用 gpt-4o 或 Claude 3.5 Sonnet。

### 坑 5：未做错误重试

网络问题或接口偶发超时，企业微信会收不到回复，用户体验差。

**解决方案：** 在调用千聚 API 的代码块里加 `try-except` 和重试逻辑，最多重试 3 次，间隔 1 秒。千聚平台可用性 99.9%，但好的开发习惯还是要的。
