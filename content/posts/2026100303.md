### 2.1 初始化客户端（避坑3：实例化方式）

官方文档可能会让你这样写：
python
import dashscope
dashscope.api_key = "你的key"
response = dashscope.Generation.call(
    model='qwen-turbo',
    prompt='你好'
)

这种写法在 `dashscope` 库V2.0版本之后，部分老API已经废弃了。而我们的写法100%兼容最新且最稳定的OpenAI标准：

python
from openai import OpenAI

client = OpenAI(
    api_key="把你的API Key粘贴到这里",
    base_url="https://www.qianjuai.com/v1"  # 就是这个！
)


**坑3核心：** 不要被老文档误导，用最新的 `openai.OpenAI` 类的 `client` 对象。所有的请求都通过 `client` 发送。

### 2.2 发起请求（避坑4：参数命名）

这是最容易出**第四个坑**的地方。官方文档的代码可能用 `prompt` 或者 `messages` 混着来。但是在通用接入里，必须严格遵守 `openai` 库的 `messages` 格式。

**错误示例（来自老文档）：**
python
# 这样写大概率会报错，因为传入的参数不标准
completion = client.completions.create(
    model="qwen-turbo",
    prompt="你好"
)


**正确写法：**
python
completion = client.chat.completions.create(
    model="qwen-turbo",
    messages=[
        {"role": "system", "content": "你是Qwen，一个有中文背景的强大助手。"},
        {"role": "user", "content": "你好"}
    ]
)


**坑4核心：** 必须用 `client.chat.completions.create` 方法，并且参数必须是 `messages` 列表格式。你不小心用了 `client.completions.create`（这是给GPT-3那种不聊天的模型用的），Qwen压根认不出来。

### 2.3 打印结果（避坑5：解析输出的格式问题）

好，代码跑完了，你以为万事大吉了？这里还有**第五个坑**。

官方文档可能会告诉你直接打印 `print(completion)`。但打印出来的是一个巨大的JSON对象，数据完全散在里面，你根本看不到纯文本结果。你得学会怎么优雅地拿到“最终回答”。

**错误示例（只打印原始对象，你找不到文本）:**
python
print(completion)


**正确操作：**
python
# 取出大模型的最终回答文本
result_text = completion.choices[0].message.content
print(result_text)


**坑5核心：** `choices[0]` 是第一个回答，`message.content` 才是文本。直接打印 `completion`，你会看到一堆 `finish_reason`、`usage`，就是找不到TEXT，然后你开始怀疑自己是不是写错了。
