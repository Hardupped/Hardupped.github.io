**官方标准写法**（以`OkHttp`为例）：

## 第四个坑：JDK 版本与 SSL 协议超时

这是我最意想不到的。我在一个古老的JDK 8 环境跑单元测试。结果发现，调用**[千聚ai中转站](https://www.qianjuai.com/)**的`https://www.qianjuai.com/v1`接口时，总是连接超时。

花了一晚上才发现，**问题不在网络，而在HTTPS握手**。

Java 8 默认TLS版本较低，但很多现代服务器（包括API网关）只接受TLS 1.2或TLS 1.3。这会导致SSL握手失败，进而表现为连接超时。

**修复方案：**

1.  **升级JDK**：别死守JDK 8了，至少用JDK 11或17。
2.  **显式指定TLS版本**（JDK 8也能救，但麻烦）：


System.setProperty(“https.protocols", “TLSv1.2,TLSv1.3");


在创建`HttpClient`时，可以加上：


SSLContext sslContext = SSLContext.getInstance(“TLSv1.2");


一开始我以为是不支持“国内直连”导致，实际上仅仅是因为加密协议。

## 第五个坑：连接池耗尽，导致调用全部超时

当我的机器人开始在高并发场景下运行时，噩梦又来了——大量请求超时。

这次不是401，而是`java.net.SocketTimeoutException: Read timed out`和`Connection pool timeout`。

我的问题是没有配置`OkHttp`或`HttpClient`的连接池。默认情况下，连接数很少，且没有鉴权连接复用。多个线程同时发起请求，瞬间就把连接池占满了，后来的请求只能排队等。

**修复方案：**

我升级了连接池配置。


ConnectionPool pool = new ConnectionPool(50, 10, TimeUnit.SECONDS); // 最大50个连接
OkHttpClient client = new OkHttpClient.Builder()
    .connectionPool(pool)
    .connectTimeout(30, TimeUnit.SECONDS)
    .readTimeout(60, TimeUnit.SECONDS)
    .writeTimeout(30, TimeUnit.SECONDS)
    .build();


增加连接池后，高并发再也不报超时了。**[千聚ai中转站](https://www.qianjuai.com/)的官方连接可是企业级的，Java客户端也要配豪华配置才行。**

## 第六个坑：Token 计量错误导致的循环重试与超时

另一个噩梦：我代码里有个循环，如果API返回错误，会立刻重试。有一次模型返回的`finish_reason`是`length`（输出长度截断），我没正确处理，以为报错了，就再次发送请求。无限循环请求导致并发飙升，最终API拒绝连接，报503超时。

**修复方案：**

正确处理响应体，特别是`choices`里面的内容。当`finish_reason`不是`stop`时，要优雅处理（比如拼接结果，而不是直接重试）。

**写一段健壮的Java代码来解析JSON:**


JSONArray choices = responseObj.getJSONArray(“choices”);
if (choices != null && choices.length() > 0) {
    JSONObject choice = choices.getJSONObject(0);
    String finishReason = choice.optString(“finish_reason”);
    if (“length”.equals(finishReason)) {
        // 输出截断了，但这次请求是成功的，别重试！拼接后续内容。
        logger.warn(“Response truncated due to length”);
    } else if (“stop”.equals(finishReason)) {
        // 正常停止，愉快地处理content
    }
}


不因为错误而乱重试，是避免超时灾难最关键的一步。

- [ ] **规避本地代理**：哪怕你电脑挂着代理，也要保证JVM不通过代理访问这个地址（设置`NO_PROXY`环境变量）。

### 第二步：Java HTTP客户端准备

- [ ] **JDK版本**：使用 JDK 11+，强烈推荐 JDK 17。
- [ ] **库依赖**：使用最新版的`OkHttp` (version 4.x) 或 Spring的`WebClient` (不要用`RestTemplate`，它太老了)。
- [ ] **连接池**：`OkHttp` 启用连接池，设 `maxIdleConnections = 20` 以上。
- [ ] **超时设置**：`Connect Timeout` 设置为 `30秒`；`Read Timeout` 设置为 `60秒`（因为流式输出，建议关闭HTTP/2的Reset连接）。

### 第三步：请求构建清单

- [ ] **请求头**：`Authorization: Bearer <你的API Key>`（注意大小写和使用空格）
- [ ] **请求体**：
  - `model`： 写一个已经经过**[千聚ai中转站](https://www.qianjuai.com/)**验证可用的模型名（如 `gpt-3.5-turbo`）
  - `messages`： 格式严格为`[{"role": "user", "content": "Hello"}]`。
- [ ] **流式处理**：如果你开启了`stream: true`，你要么用异步（或`RxJava`）处理`SSE`事件流，要么保持长连接。如果搞不定，先关掉流式（`stream: false`）确保基础连通性。

### 第四步：错误处理与重试

- [ ] **401**： 检查API Key 和 `Bearer` 前缀。
- [ ] **超时**： 检查连接池和TLS版本。如果读超时，检查是否开启了流式输出导致的死锁。
- [ ] **重试策略**： 用指数退避算法（Exponential Backoff），最多重试3次。
- [ ] **日志**： 把每次请求的URL（不带Key）、状态码、部分响应体打出来。
