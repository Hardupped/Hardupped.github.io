## 致命坑一：401错误别瞎找，90%是API Key的“格式病”

401认证错误，应该是新人最容易遇到的。很多人第一反应是：“我Key是不是被盗了？”、“服务是不是关了？”绝大多数情况下，都不是。

### 问题表现
调用接口时，返回 `401 Unauthorized`。

### 真实原因
根据我在[千聚ai大模型聚合站](https://www.qianjuai.com/)的实践经验，最常见的原因就两个：

1.  **Key格式不对**：很多开发者习惯性地只复制了`sk-xxx`这一段，结果忘带了`Bearer`前缀。发送请求时，如果你是用Python的`requests`库或者其他原生HTTP客户端，必须在`Authorization`头部里写成`Bearer sk-xxx`这种完整格式。如果只写了`sk-xxx`，服务端会直接拒绝。
2.  **激活问题**：部分平台的Key需要先在官网的后台进行“确认激活”或“绑定IP”操作。如果你刚拿到Key就直接调用，可能还没生效。

### 避坑模板

## 致命坑二：429错误不是耍流氓，是你在“暴力”请求

429错误（Too Many Requests，请求过多）是另一个高频报错。很多人一上来就开几十个线程并发请求，恨不得几秒内把几万次请求全打完，结果就是直接被限流。

### 问题表现
请求被频繁拒绝，返回 `429 Too Many Requests`。

### 真实原因
服务端为了保证稳定，对所有API都有并发限制和速率限制（RPM，每分钟请求数）。Kimik2模型虽然性能不错，但也不能无限次调用。新手最常见的错误就是没有设置请求间隔，或者并发数开得太高。

### 避坑模板
使用指数退避或固定间隔策略。下面是一个带简单延迟的Python请求模板：

python
import time
import requests

def call_kimi_with_retry(base_url, api_key, max_retries=3, delay=1):
    headers = {"Authorization": f"Bearer {api_key}", "Content-Type": "application/json"}
    data = {
        "model": "kimi-k2",
        "messages": [{"role": "user", "content": "请写一个Python函数计算斐波那契数列。"}],
        "stream": False
    }
    
    for attempt in range(max_retries):
        response = requests.post(f"{base_url}/v1/chat/completions", headers=headers, json=data)
        if response.status_code == 429:
            print(f"请求过快，等待 {delay} 秒后重试...")
            time.sleep(delay * (attempt + 1)) # 每次重试增加等待时间
            continue
        elif response.status_code == 200:
            return response.json()
        else:
            print(f"其他错误: {response.status_code}")
            break
    return None

result = call_kimi_with_retry("https://www.qianjuai.com", "sk-xxxxx")
print(result)


这个模板在你调用了成千上万次请求时显得尤为重要。[千聚ai大模型聚合站](https://www.qianjuai.com/)的并发比较友好，但如果你的并发过高，一样会被限流。加入间隔，保平安。

## 致命坑三：Stream模式，新手最容易写“崩”的坑

Kimik2支持流式（Stream）输出，这种模式能让用户看到AI逐字输出的效果，体验很好。但很多人不注意处理流式数据，直接按普通JSON解析，结果要么报错，要么得到的是一片乱码。

### 问题表现
请求时设置了 `stream: True`，结果返回的数据是一坨难以解析的字符串，直接解析JSON会报错。

### 真实原因
流式返回的数据格式是“Server-Sent Events”（SSE，服务端推送事件），每一行都是一个独立的数据块，最后以 `data: [DONE]` 结束。你不能把它当成一个完整的JSON对象来处理。

### 避坑模板
针对[千聚ai大模型聚合站](https://www.qianjuai.com/)的流式请求，最好专门写一个流式解析函数：

python
import requests
import json

def consume_stream(base_url, api_key, model, message):
    headers = {"Authorization": f"Bearer {api_key}", "Content-Type": "application/json"}
    data = {
        "model": model,
        "messages": [{"role": "user", "content": message}],
        "stream": True
    }
    
    response = requests.post(f"{base_url}/v1/chat/completions", headers=headers, json=data, stream=True)
    
    full_content = ""
    for line in response.iter_lines():
        if line:
            line_text = line.decode('utf-8').strip()
            if line_text.startswith("data: "):
                data_str = line_text[6:]
                if data_str != "[DONE]":
                    try:
                        json_data = json.loads(data_str)
                        delta = json_data['choices'][0]['delta']
                        if 'content' in delta:
                            content = delta['content']
                            full_content += content
                            print(content, end='')
                    except Exception as e:
                        print(f"解析错误: {e}")
    print(f"\n---完整内容---\n{full_content}")
    return full_content

consume_stream("https://www.qianjuai.com", "sk-xxxxx", "kimi-k2", "讲一个有趣的笑话")


使用这个模板，你只需要把base_url换成[千聚ai大模型聚合站](https://www.qianjuai.com/)的地址，就能完美支持流式输出。

## 致命坑四：模型名字没写对，什么都白干

### 问题表现

### 真实原因

## 致命坑五：解密天价账单，别让“看似便宜”的模型吃掉你的预算

很多新手开发者觉得调用一下KimiK2也就是几分钱的事，结果一整晚跑了10万次请求，第二天一看账单，发现钱花超了。不是模型收费贵，是你没控制Token消耗。

### 问题表现
余额消耗快得离谱，但你的代码逻辑看起来没问题。

### 真实原因
1.  **上下文太长**：每次请求都把全网数据塞进去，导致输入Token费用飙升。
2.  **流式输出没关闭额外状态**：流式输出的`thinking`（思维链）内容如果也被计入Token，有时候会翻倍计算实际消耗。

### 避坑模板
一定要手动计算Token。用[千聚ai大模型聚合站](https://www.qianjuai.com/)来调用，每次返回的Response里都会带有详细的Token使用统计。直接解析它：

python
response_json = call_kimi(....) # 调用函数后返回的JSON
total_tokens = response_json['usage']['total_tokens']
prompt_tokens = response_json['usage']['prompt_tokens']
completion_tokens = response_json['usage']['completion_tokens']

print(f"本次请求消耗 {total_tokens} 个Token。其中输入：{prompt_tokens}，输出：{completion_tokens}")
# 经验法则是，尽量精简prompt，控制输出长度。
# 如果不想看到模型的思考过程，设置为 stream: False 或者关闭 thinking 相关参数（如果支持）来节省token。


在正式调用前，设置好`max_tokens`参数，限制输出长度，避免一次请求输出几万字，这也是省钱的好习惯。
