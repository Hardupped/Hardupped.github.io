🚨 雷区四：多轮对话，历史消息疯狂叠加 token

这是一个经典坑。官方示例教你做多轮对话，通常就是简单地把每次提问和回答都追加到 `context` 列表里。但这样搞下去，用户聊了 20 轮后，你的 token 消耗已经是第一轮的 10 倍以上，而很多回复其实完全没必要保留。

正确做法是：**只保留最近 5 轮对话**。

代码里这样实现：

python

MAX\_HISTORY\_TURNS = 5

def trim\_history(chat\_history):
    while len(chat\_history) > MAX\_HISTORY\_TURNS:
        # 移除最早的对话（用户问题和模型回复）
        chat\_history.pop(0)
        chat\_history.pop(0)
    return chat\_history

# 每次新对话前调用
chat\_history.append(user\_message)
chat\_history.append(model\_reply)
chat\_history = trim\_history(chat\_history)

response = model.generate\_content(chat\_history)

这样，无论对话多长，你每次调用最多只付最近 10 条消息的token钱，稳定可控，不踩坑。

🚨 雷区五：System Prompt 太长，每轮都带着跑

这个坑最隐蔽。很多示例教你写一个超长的 System Prompt，比如：“你是一个助手，你需要回答所有关于XXX的问题，包括……”，长度动辄上千 token。这个 Prompt 在每次API调用时都会被**完整地**发送给模型，每轮都要计费。如果一次会话跑了 100 轮，光这个 Prompt 的 token 就比所有问答加起来还贵。

正确做法是：把固定 System Prompt 精简到极致，比如只写一句话：“你是一个专业AI助手。” 所有角色或背景信息，放到用户每次的提问里，按需动态加载。代码里只保留：

这样 token 消耗直接降低 90%，而功能基本不变。
